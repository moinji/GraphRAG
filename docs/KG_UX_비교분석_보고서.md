## A. WEF Strategic Intelligence — Transformation Map 분석

### A-1. 장점

**1. 압도적 시각적 임팩트 (Visual Impact)**
방사형 원형 레이아웃은 처음 보는 순간 "이건 단순한 목록이 아니다"라는 인식을 즉시 심어준다. 중심 토픽에서 방사형으로 퍼지는 구조가 시스템 사고(Systems Thinking)를 시각적으로 체현하며, 경영진 프레젠테이션이나 보고서 임베드 시 강력한 설득력을 갖는다.

**2. 비선형 탐색의 자연스러운 유도**
하나의 이슈(예: "해양 거버넌스")를 경제, 기술, 환경 등 다양한 렌즈로 접근할 수 있는 구조가 토픽 간 연결선으로 자연스럽게 표현된다. 사용자가 "나는 지금 관점을 전환하고 있다"는 사실을 시각적으로 인지할 수 있어, 탐색 자체가 학습 경험이 된다.

**3. 콘텐츠-그래프 통합 (Content-Graph Fusion)**
노드를 클릭하면 관련 기사, 리포트, 데이터 시각화, 전문가 목록이 피드로 제공된다. 그래프가 "구조"만 보여주는 것이 아니라 "실체"(콘텐츠)까지 연결하므로, 사용자가 그래프에서 얻은 구조적 인사이트를 즉시 심화할 수 있다.

**4. AI + 전문가 큐레이션 하이브리드**
Semantic fingerprinting과 graph embedding으로 자동 진화하는 KG 위에, 대학/싱크탱크 전문가가 큐레이션을 더한다. 이 하이브리드 접근이 "기계가 놓치는 맥락"과 "사람이 처리 못하는 규모"를 동시에 해결한다.

**5. 임베드/공유 생태계**
위젯으로 외부 사이트에 삽입 가능하고, 커스텀 맵 생성 기능으로 개인/조직별 맞춤 뷰를 만들 수 있다. 이는 단일 도구를 넘어 "플랫폼"으로의 확장성을 보여준다.

**6. 정보설계의 계층화**
글로벌 이슈 → 하위 토픽 → 개별 콘텐츠의 3단계 계층이 명확하다. 사용자가 현재 어느 추상화 수준에 있는지 항상 인지할 수 있다.

### A-2. 단점

**1. 높은 초기 학습곡선**
250+ 토픽이 방사형으로 펼쳐진 화면을 처음 보는 사용자는 "어디서부터 시작해야 하지?"라는 인지적 마비(cognitive paralysis)를 경험할 가능성이 높다. 명확한 온보딩 가이드나 추천 경로가 없으면 이탈률이 높아진다.

**2. 정보 과부하 (Information Overload)**
토픽 간 연결선이 많아질수록 "스파게티 다이어그램" 현상이 발생한다. 수천 개의 가능한 탐색 경로가 존재하며, 이는 탐색의 자유도가 아니라 의사결정의 부담으로 작용할 수 있다.

**3. 관계의 의미론적 빈곤**
토픽 A와 B가 "관련있다"는 것은 알 수 있지만, "어떻게" 관련있는지(인과, 의존, 상충, 시간적 선후)는 연결선만으로 구분이 안 된다. 모든 엣지가 동질적으로 보이는 것은 KG 시각화의 치명적 약점이다.

**4. 기업 데이터 직접 적용의 어려움**
WEF의 노드는 "토픽"(추상적 개념)이지만, 기업 데이터의 노드는 "고객 ID 1234", "주문 #5678" 같은 구체적 인스턴스다. 250개의 글로벌 토픽을 원형으로 배치하는 것과, 50,000개의 고객 레코드를 시각화하는 것은 본질적으로 다른 문제다.

**5. 정적 탐색 구조**
사용자의 질문이나 맥락에 따라 레이아웃이 동적으로 변하지 않는다. "AI가 공급망에 미치는 영향을 보여줘"라고 하면 해당 경로가 하이라이트되어야 하는데, 현재는 사용자가 수동으로 노드를 찾아 클릭해야 한다.

**6. 모바일 경험 한계**
방사형 레이아웃은 데스크탑의 넓은 화면에 최적화되어 있으며, 모바일에서는 핀치 줌과 팬이 빈번해져 탐색 피로도가 급격히 증가한다.

### A-3. 기업 데이터 적용 시 Gap

| 차원 | WEF (토픽) | 기업 데이터 (엔티티) | Gap |
|------|-----------|---------------------|-----|
| 노드 수 | ~250개 (관리 가능) | 수만~수백만 (집계 필수) | 집계/샘플링 전략 필요 |
| 노드 본질 | 추상 개념 | 구체 인스턴스 | 개별 레코드 수준 탐색 필요 |
| 관계 유형 | "관련있음" 단일 유형 | PLACED, CONTAINS 등 다종 | 다중 엣지 타입 시각화 필요 |
| 갱신 주기 | 일/주 단위 | 실시간~분 단위 | 실시간 업데이트 UX 필요 |
| 사용자 목적 | 거시 트렌드 이해 | 운영 의사결정, 이상 탐지 | 액션 연결(alert, drill-down) 필요 |

## B. GraphRAG 프로젝트 — Cytoscape 기반 KG Explorer 분석

### B-1. 장점

**1. 완전한 E2E 파이프라인**
DDL 업로드 → 온톨로지 자동 생성 → 사람 리뷰 → KG 빌드 → Q&A + 시각화까지 끊김 없이 연결된다. 이는 PoC를 넘어 실제 제품의 골격을 갖추고 있다는 점에서 기술적 완성도가 높다.

**2. Human-in-the-Loop 온톨로지 리뷰**
FK 규칙 기반 자동 생성 후 사람이 검증/수정하는 워크플로우는 "자동화의 함정"(잘못된 온톨로지가 전파되는 문제)을 원천 차단한다. 기업 환경에서 신뢰성 확보에 핵심적인 설계다.

**3. Cytoscape.js의 기술적 견고함**
캔버스 기반 렌더링으로 1000+ 노드를 무리 없이 처리하고, 15+ 레이아웃 알고리즘을 제공한다. Lazy expansion(이웃 노드 온디맨드 로딩)은 대규모 그래프에서의 성능을 보장한다.

**4. 하이브리드 쿼리 라우팅**
규칙 기반(빠름, 정확) + LLM 기반(유연, 자연어) 쿼리 라우팅을 결합한 것은 실용적인 설계다. 템플릿 매칭이 안 되면 LLM이 직접 Cypher를 생성하는 폴백 구조가 있다.

**5. 타입별 색상 코딩 + 필터링**
12종 노드 타입에 고유 색상을 부여하고, 좌측 사이드바에서 토글 가능한 것은 복잡한 그래프에서 빠른 시각적 구분을 가능하게 한다.

**6. 증거 경로(Evidence Path) 표시**
Q&A 응답 시 단순 텍스트 답변이 아니라 근거가 되는 그래프 경로를 함께 보여주는 것은 신뢰성 확보에 중요한 차별점이다.

### B-2. 단점

**1. 시각적 임팩트 부족**
COSE(force-directed) 레이아웃은 기능적으로는 유효하지만, 처음 보는 비기술 사용자에게 "노드와 선이 엉킨 복잡한 그림"으로 인식될 수 있다. WEF의 방사형 레이아웃이 주는 "아, 이건 뭔가 다르다"라는 첫인상이 없다.

**2. 탐색 내러티브의 부재**
"이 그래프로 무엇을 할 수 있는가?"에 대한 가이드가 없다. 사용자가 그래프를 보고 스스로 의미 있는 탐색 경로를 발견해야 하는데, 이는 도메인 지식이 없는 사용자에게는 매우 어렵다.

**3. 인사이트 도출 수동성**
현재 UX는 "데이터를 보여준다"에 그친다. "이 고객은 비정상적으로 많은 반품을 했다", "이 공급업체는 3개 카테고리의 유일한 소스다" 같은 인사이트를 시스템이 선제적으로 제시하지 않는다.

**4. Q&A와 그래프 탐색의 분리**
Query 페이지(채팅)와 Explore 페이지(그래프)가 별개 화면이다. "가장 많이 주문한 고객은?"이라는 질문의 답이 그래프에서 즉시 하이라이트되는 연동이 없어, 두 기능의 시너지가 미약하다.

**5. 비기술 사용자 진입 장벽**
DDL 업로드, 온톨로지 리뷰, Cypher 기반 Q&A는 모두 기술적 배경 지식을 전제한다. C레벨 임원이나 비즈니스 분석가가 독립적으로 사용하기 어렵다.

**6. 콘텐츠 연동 부재**
노드가 가진 것은 DB 속성(이름, 이메일, 가격)뿐이다. WEF처럼 관련 리포트, 뉴스, 트렌드 등 외부 콘텐츠와 연결되어 있지 않아, 그래프 탐색의 "끝"이 속성 확인에서 멈춘다.

### B-3. WEF 대비 부족한 점

| 관점 | WEF | GraphRAG | 개선 방향 |
|------|-----|----------|----------|
| 정보설계 | 토픽→하위토픽→콘텐츠 3계층 | 노드→속성 2계층 | 노드→관계패턴→비즈니스인사이트 계층 추가 |
| 콘텐츠 연동 | 250+ 소스의 기사/리포트 피드 | DB 속성만 표시 | 관련 문서/메모/외부 데이터 연동 |
| 내러티브 | 비선형이지만 "관점 전환"이라는 스토리 | 없음 | 가이드된 탐색 경로 제공 |
| 사용자 여정 | 호기심 → 탐색 → 심화 → 공유 | 업로드 → 빌드 → 조회 | "발견"의 모멘트를 설계해야 함 |

## D. 제3의 UX — 차세대 KG 시각화 제안

### 제안 1: Semantic Zoom Dashboard ("줌-어스" 패턴)

**컨셉:** Google Earth처럼 줌 레벨에 따라 추상화 수준이 자동으로 변하는 KG 탐색. 멀리서 보면 카테고리 클러스터, 가까이 가면 개별 엔티티와 속성이 나타난다.

**핵심 인터랙션 패턴:**
1. 초기 화면: 전체 스키마를 노드 타입 단위로 표시 (각 노드에 총 개수 배지)
2. 마우스 휠/핀치 줌: 노드 타입 → 클러스터(VIP/일반/신규) → 개별 엔티티로 점진적 전환
3. 각 줌 레벨에서 해당 수준의 집계 통계가 자동 표시
4. 하단 브레드크럼으로 현재 탐색 경로 추적 + 뒤로가기

**WEF/GraphRAG 대비 차별점:**
- WEF: 모든 토픽이 동일 레벨 → Semantic Zoom은 자연스러운 계층 탐색
- GraphRAG: 500개 노드 일괄 표시 → Semantic Zoom은 수만 노드도 인지 부하 없이 탐색

**구현 난이도:** Medium
- 추천 스택: Cytoscape.js의 compound nodes + 커스텀 줌 이벤트 핸들러
- 백엔드: `/api/v1/graph/aggregate?level=type|cluster|entity` 엔드포인트 추가
- Neo4j 쿼리: `MATCH (n:Customer) RETURN n.segment, count(*) GROUP BY n.segment`

**기업 시나리오:**
"COO가 공급망 전체를 조감하다가, 특정 카테고리의 공급업체 집중도가 비정상적으로 높은 것을 발견하고, 줌인하여 해당 공급업체 3곳의 구체적 거래 내역까지 드릴다운하는 흐름"

### 제안 2: Conversational Graph Navigator ("대화형 그래프")

**컨셉:** 자연어 질문을 하면 그래프가 실시간으로 반응하여 관련 영역이 하이라이트되고, 답변이 그래프 위에 오버레이되는 통합 경험.

```
┌──────────────────────────────────────────────────────────┐
│  GraphRAG Explorer                          [Settings]   │
├─────────────────────────┬────────────────────────────────┤
│                         │                                │
│    Interactive Graph     │   Conversation Panel           │
│    (Cytoscape Canvas)   │                                │

│   │   │    │   │       │   Bot: Customer_42 (Kim)이      │
│   └───┘    └─┬─┘       │   23건으로 가장 많습니다.         │
│       +======+======+   │                                │
│       |  * C_42     |   │   Evidence Path:               │
│       | (하이라이트) |   │   Customer_42 --PLACED-->      │
│       |  23건 반품   |   │   Order_* --HAS_RETURN-->      │
│       +=============+   │   Return (23건)                │
│              |          │                                │

│         │  (23)   │     │   ────────────────────────      │

│                         │                                │
│  [필터] [레이아웃]       │   Bot: 주로 전자제품 카테고리에서  │
│                         │   월 평균 4.3건 주문하며...       │
│                         │                                │
```

**핵심 인터랙션 패턴:**
1. 좌측 그래프 + 우측 채팅이 동일 화면에 공존
2. 자연어 질문 → LLM이 Cypher 생성 → 결과 노드/엣지가 그래프에서 하이라이트 + 줌
3. 채팅 답변에 "그래프에서 보기" 버튼 → 클릭 시 해당 서브그래프로 자동 포커스
4. 그래프에서 노드 클릭 → 채팅에 자동으로 "이 노드에 대해 더 알려줘" 질문 생성
5. 하단 인사이트 바: 시스템이 선제적으로 이상치/패턴을 감지하여 제안

**WEF/GraphRAG 대비 차별점:**
- WEF: 탐색은 수동 클릭만 가능 → 대화형은 자연어로 의도 표현
- GraphRAG: Query/Explore 페이지 분리 → 하나의 통합 화면에서 질문과 시각화가 실시간 연동

**구현 난이도:** Medium-High
- 추천 스택: 기존 Cytoscape + QueryPage 통합, WebSocket으로 실시간 하이라이트
- 핵심 변경: QueryPage와 ExplorePage 병합, `highlightNodes(nodeIds[])` 함수 추가
- 신규: LLM 응답에서 관련 node ID 추출 → Cytoscape `cy.getElementById()` 하이라이트

**기업 시나리오:**
"구매팀 매니저가 '지난 분기에 납기 지연이 가장 많았던 공급업체는?'이라고 질문하면, 그래프에서 해당 공급업체 노드가 빨간색으로 빛나며, 연결된 주문과 제품까지 경로가 표시되고, 하단에 '이 공급업체의 지연률은 업계 평균 대비 3.2배입니다'라는 인사이트가 자동 제시되는 흐름"

- 신규 컴포넌트: StoryCard, StoryPlayer, StoryStep (프레젠테이션 모드)
- 백엔드: `/api/v1/insights/detect` (이상치/패턴 탐지), `/api/v1/stories/generate`

## E. 실행 가능한 로드맵 — "Conversational Graph Navigator" 선정

**선정 이유:** 기존 코드베이스(QueryPage + ExplorePage)를 병합하는 것이므로 신규 개발 범위가 가장 작고, 데모에서 "질문하면 그래프가 반응한다"는 와우 팩터가 가장 즉각적이다.

### E-1. 구현 방안

```
현재 구조:
  QueryPage.tsx (채팅)  <-->  ExplorePage.tsx (그래프)  [별개 페이지]

목표 구조:
  UnifiedExplorerPage.tsx
  ├── GraphCanvas (좌측 60%)
  ├── ChatPanel (우측 40%)
  ├── InsightBar (하단)
  └── use-graph-chat-sync.ts (연동 훅)
```

### E-2. 필요한 신규 라이브러리/기술

| 라이브러리 | 용도 | 비고 |
|-----------|------|------|
| 없음 (기존 스택 활용) | Cytoscape + React + FastAPI | 신규 의존성 최소화 |
| `framer-motion` (선택) | 하이라이트 애니메이션 | 이미 shadcn/ui 생태계와 호환 |

### E-3. 변경해야 할 기존 컴포넌트

| 컴포넌트 | 변경 내용 |
|----------|----------|
| `App.tsx` | 새 라우트 `/unified` 추가 |
| `QueryPage.tsx` | ChatPanel로 분리 추출 (사이드 패널 형태) |
| `ExplorePage.tsx` | UnifiedExplorerPage로 진화 (ChatPanel 통합) |
| `use-graph-data.ts` | `highlightNodes(ids[])`, `focusOnSubgraph(ids[])` 함수 추가 |
| `graph-styles.ts` | 하이라이트 스타일 추가 (glow effect, pulse animation) |
| `executor.py` (백엔드) | 응답에 `related_node_ids: string[]` 필드 추가 |
| `client.ts` (API) | query 응답에서 node ID 배열 파싱 |

### E-4. 예상 작업량

| 작업 단위 | 예상 파일 수 | 핵심 내용 |
|-----------|-------------|----------|
| ChatPanel 컴포넌트 분리 | 2 | QueryPage에서 채팅 UI를 독립 컴포넌트로 추출 |
| UnifiedExplorerPage | 1 | 좌우 분할 레이아웃 + 상태 통합 |
| use-graph-chat-sync 훅 | 1 | 채팅 응답 → 그래프 하이라이트 연동 로직 |
| 하이라이트 스타일/애니메이션 | 1 | Cytoscape 노드 glow, edge path 강조 |
| 백엔드 node ID 반환 | 2 | executor에서 관련 노드 ID 추출 + 응답 스키마 확장 |
| InsightBar (선택) | 1 | 하단에 자동 감지된 이상치 표시 |
| **총계** | **~8 파일** | — |

### E-5. 데모 와우 팩터 — "질문하면 그래프가 살아난다"

**핵심 기능 1가지:**

> **Live Query Highlighting** — 사용자가 "가장 많이 주문한 고객 상위 3명은?"이라고 질문하면, 채팅에 텍스트 답변이 나오는 동시에 그래프 캔버스에서 해당 3개 Customer 노드가 **펄스 애니메이션**으로 빛나고, 그들의 Order 연결선이 **굵은 컬러 라인**으로 강조되며, 캔버스가 자동으로 해당 영역으로 **스무스 줌인**된다.

이 단 하나의 기능만으로도 "정적 그래프 뷰어"가 "지능형 데이터 탐색 도구"로 인식이 전환된다.

## 종합 결론

| 항목 | 핵심 요약 |
|------|----------|
| **WEF 강점** | 시각적 임팩트 + 콘텐츠 연동 + 비선형 탐색 |
| **GraphRAG 강점** | E2E 파이프라인 + 기술적 확장성 + 커스터마이징 자유도 |
| **가장 큰 Gap** | Q&A와 시각화의 분리, 인사이트 도출의 수동성 |
| **최우선 개선** | Query + Explore 통합 → Conversational Graph Navigator |
| **장기 비전** | Semantic Zoom + Narrative Explorer를 순차적으로 추가 |

> 가장 빠르게 격차를 줄이는 방법은 **Query와 Explore를 하나로 통합**하여, 질문이 곧 시각화가 되는 경험을 만드는 것이다.
