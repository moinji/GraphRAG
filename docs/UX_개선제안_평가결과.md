### A-1. 기대 vs 현실 Gap 분석

#### 제안 1: Semantic Zoom Dashboard

### A-2. 숨겨진 기술적 함정 (Hidden Pitfalls)

#### 제안 1: Semantic Zoom

1. **클러스터링 기준의 부재.** 현재 Neo4j 노드에는 `segment`, `tier`, `category_group` 같은 클러스터 속성이 없다. "VIP/일반/신규" 분류를 하려면 먼저 비즈니스 로직(주문 횟수 기준? 매출 기준?)을 정의하고 Neo4j에 속성을 추가해야 한다. 이것은 UX 문제가 아니라 **데이터 모델링 문제**다.

2. **Cytoscape compound nodes의 줌 전환 UX.** Cytoscape의 compound node는 "부모 노드 안에 자식 노드가 들어있는" 구조지, "줌하면 자동으로 전환되는" 구조가 아니다. 줌 이벤트를 감지해서 노드를 삭제/추가/레이아웃 재계산하는 로직을 직접 구현해야 하는데, 이 과정에서 **레이아웃 점핑**(노드 위치가 갑자기 바뀌는 현상)이 발생한다.

3. **줌 레벨 경계의 애매함.** 줌 레벨 1→2 전환 시점을 어떻게 결정하는가? 줌 배율 2.0x? 3.0x? 사용자마다 다른 화면 크기에서 같은 줌 배율이 다르게 느껴진다. 이 "전환 임계값" 튜닝이 예상보다 오래 걸리고, 잘못 설정하면 줌할 때마다 의도치 않은 레벨 전환이 발생한다.

#### 제안 2: Conversational Graph Navigator

1. **executor.py의 벽.** 제안서는 "~8 파일 변경"이라고 했지만, 핵심인 "자연어 → Cypher → node ID 반환"이 현재 **존재하지 않는다.** executor.py를 범용으로 만드는 것은 프론트엔드 병합보다 3-5배 큰 작업이다. 이것 없이 프론트엔드만 병합하면 **빈 껍데기**가 된다.

2. **하이라이트 대상 노드가 로드되지 않았을 수 있음.** 쿼리 결과로 나온 Customer_42가 현재 캔버스의 500개 노드 안에 없을 수 있다. 이 경우 (a) 해당 노드를 추가 로드? (b) 없다고 알림? (c) 전체 그래프를 다시 로드? 어떤 선택이든 추가 복잡도가 크고, 제안서에서 이 케이스를 다루지 않았다.

3. **채팅 패널의 화면 비율 경쟁.** 좌측 60% + 우측 40%는 1920px 모니터에서 그래프 영역이 ~1150px → 기존 ExplorePage의 캔버스보다 좁아진다. 좌측에 필터 사이드바까지 있으면 실제 그래프 영역은 ~850px. 이는 500개 노드 렌더링 시 기존보다 **더 좁고 혼잡한** 경험을 만든다.

#### 제안 3: Narrative Explorer

1. **샘플 데이터의 본질적 한계.** GraphRAG의 데이터는 `generator.py`가 FK 규칙에 따라 생성한 합성 데이터다. "공급망 집중 위험"이나 "VIP 이탈 징후" 같은 인사이트는 **실제 비즈니스 패턴이 반영된 데이터에서만 의미가 있다.** 랜덤 생성된 데이터에서 LLM이 찾는 "패턴"은 통계적 노이즈일 뿐이다.

2. **스토리 정확성 검증 불가.** LLM이 "Supplier Alpha가 유일한 공급원"이라고 생성했는데 실제로는 아닌 경우, 사용자가 이를 검증할 방법이 없다. 잘못된 인사이트를 그럴듯하게 포장하는 것은 "없는 것보다 나쁜" 결과를 만든다.

3. **스토리 간 연결의 부재.** 스토리 카드가 독립적이면 "공급망 위험"과 "VIP 이탈"이 실제로는 연관된 현상(같은 공급업체의 품질 문제가 VIP 고객 이탈 원인)임을 보여줄 수 없다. 스토리 간 관계를 모델링하면 또 다른 KG가 필요해지는 재귀적 복잡도가 생긴다.

### A-3. 과대평가 vs 과소평가

| 제안 | 과대평가 | 과소평가 |
|------|---------|---------|
| **1. Semantic Zoom** | "수만 노드도 인지 부하 없이" — 클러스터링 기준이 없으면 의미 없는 그룹핑이 될 뿐 | 줌 레벨 전환 애니메이션의 품질이 UX 성패를 결정. 끊기면 오히려 역효과 |
| **2. Conversational** | "~8 파일 변경" — 프론트 병합은 맞지만 범용 쿼리 백엔드가 빠져있음 | 노드 클릭 → 자동 질문 생성의 가치. 이것만 해도 탐색 경험이 크게 개선됨 |
| **3. Narrative** | "비기술 사용자 즉시 가치" — 샘플 데이터에서의 스토리는 가치가 없음 | StoryPlayer UI 자체의 범용성. 스토리 생성을 수동(분석가 작성)으로 해도 가치 있음 |

**근거:**
- 제안 2가 1위인 이유: "질문하면 그래프가 반응"은 30초 데모에서 가장 직관적이고, 범용 쿼리 파이프라인이 완성되면 모든 후속 기능의 기반이 됨

- 제안 3이 3위인 이유: 컨셉은 최고지만 (a) 구현 비용 최대 (b) 샘플 데이터에서는 거짓 인사이트 리스크 (c) 실 데이터가 들어오는 시점에야 진가 발휘

## B. 기존 제안의 개선안

### 제안 1 개선안: Semantic Zoom → "Schema-First Overview"

기존 문제: 클러스터링 기준 부재, 줌 전환 UX 난이도
핵심 전환: "3단계 줌"을 포기하고, **"스키마 개요 → 인스턴스 탐색" 2단계**로 단순화

**MVP 범위 (1주 이내):**
- ExplorePage 진입 시 첫 화면을 **스키마 뷰**(노드 타입 + 카운트 + 관계선)로 변경
- 기존 `/graph/stats` 응답을 활용하여 노드 타입별 원 크기를 카운트에 비례시킴
- 노드 타입 클릭 → 해당 타입의 인스턴스만 필터링하여 기존 COSE 레이아웃으로 전환
- 상단에 브레드크럼: `전체 스키마 > Customer (500)` 형태

**MVP에서 제거한 것과 이유:**
- 중간 레벨(클러스터): 클러스터링 기준이 없으므로 삭제. 실 데이터 + 비즈니스 로직이 있을 때 추가
- 줌 기반 전환: 클릭 기반 전환으로 변경. 줌 임계값 튜닝 불필요

**MVP만으로 전달되는 핵심 가치:**
"그래프를 처음 보는 사용자가 500개 노드에 압도당하지 않고, 스키마 구조를 먼저 이해한 뒤 관심 있는 엔티티 타입으로 진입한다"

**v1 확장 (2-3주):** 스키마 뷰에서 관계선 클릭 → 해당 관계 타입의 인스턴스 쌍을 리스트로 표시
**v2 확장 (1개월+):** 노드 속성 기반 자동 클러스터링 (금액 구간, 날짜 구간 등)

### 제안 2 개선안: Conversational Graph → "Query-Linked Explorer"

기존 문제: executor.py 데모 전용, 범용 쿼리 미구현, 화면 비율 경쟁
핵심 전환: 풀 통합 대신 **"쿼리 결과를 그래프에서 보기" 단방향 링크**로 시작

**MVP 범위 (1주 이내):**
- ExplorePage에 **미니 쿼리 바** 추가 (상단, 검색바 옆에 배치. 별도 패널 아님)
- 기존 5개 데모 질문을 드롭다운으로 제공 (자유 입력은 v1에서)
- 데모 질문 선택 → 기존 executor 호출 → 응답의 answer 텍스트를 **토스트/배너로 표시**
- **핵심**: executor.py 응답에 `related_node_ids` 필드 추가 (3개 데모 질문에 대해 하드코딩)
- 해당 node ID를 Cytoscape에서 하이라이트 (노란 보더 + 줌인 — 기존 검색 하이라이트 로직 재사용)

**MVP에서 제거한 것과 이유:**
- 좌우 분할 레이아웃: 그래프 영역을 줄이지 않기 위해 삭제. 미니 바로 대체
- 범용 자연어 입력: executor가 데모 전용이므로 불가. 드롭다운으로 한정
- WebSocket: 불필요. 기존 fetch + 상태 업데이트로 충분
- InsightBar: v2로 이관

**MVP만으로 전달되는 핵심 가치:**

기존 문제: 샘플 데이터 무의미, LLM 정확성 미검증, 구현 비용 최대

**MVP만으로 전달되는 핵심 가치:**
"그래프를 열면 이미 의미있는 발견 3가지가 카드로 나와 있다. 클릭하면 해당 영역이 하이라이트된다." — 능동적 탐색 없이도 가치를 전달

## C. 제4의 대안 제안

### 기존 3가지 제안이 공유하는 맹점

1. **"실시간 인터랙티브 탐색"에만 집중.** 멘토에게 보여줄 때 중요한 것은 "이 도구를 어떤 비즈니스 맥락에서 쓰는가"인데, 3가지 모두 "그래프를 더 잘 만지작거리는 법"만 제안한다.

2. **단일 사용자 시나리오만 가정.** 현실의 B2B SaaS에서는 분석가가 발견한 것을 팀장에게 공유하고, 팀장이 경영진에게 보고하는 흐름이 있다. 3가지 모두 "혼자 탐색"만 다룬다.

3. **"그래프 진입 전" 사용자 여정 무시.** 사용자가 왜 이 도구에 접속했는지, 무엇을 찾고 싶은지에 대한 "의도 파악" 단계가 없다. 곧바로 복잡한 그래프를 보여주는 것은 "빈 검색창에 뭘 검색해야 할지 모르는" 문제와 같다.

### Phase 3 (5-8주): "Insight Cards + Snapshot & Share + 좌우 통합"

**목표:** 시스템이 선제적으로 인사이트를 제시하고, 발견을 공유할 수 있는 협업 도구로 진화

**추가되는 것:**

## 핵심 결론

> **프론트엔드 UX 개선보다 백엔드 쿼리 파이프라인 완성이 먼저다.**

기존 3가지 제안 모두 "화면을 어떻게 바꿀까"에 집중했지만, 현실은 **executor.py가 3개 데모 질문만 처리하는 상태**에서 아무리 화려한 프론트엔드를 만들어도 빈 껍데기가 된다는 것이다.

Phase 1에서 하드코딩으로 "맛보기"를 빠르게 만들고, Phase 2에서 범용 쿼리 파이프라인을 구축하는 것이 **가장 현실적이면서도 임팩트 있는 경로**다.

그리고 기존 제안에 없던 **Dashboard-First 진입점**과 **Snapshot & Share**가 실제 B2B 사용 맥락에서는 "그래프 시각화 개선"보다 더 높은 사용자 가치를 제공할 수 있다.

**멘토 데모 적정 시점:** Phase 1 완료 직후. Dashboard + 하이라이트 연동만으로 "방향성이 보인다"는 피드백을 받을 수 있고, Phase 2 작업에 대한 멘토의 우선순위 의견을 반영할 수 있다.
