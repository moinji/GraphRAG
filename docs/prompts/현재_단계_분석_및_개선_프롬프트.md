# GraphRAG 현재 단계 분석 및 개선 프롬프트

> 이 프롬프트를 실행하면 현재 프로젝트를 7축으로 분석하고, 코드 레벨의 개선 계획을 생성합니다.

---

## 역할

당신은 B2B SaaS 아키텍트이자 시니어 풀스택 엔지니어입니다.
GraphRAG 프로젝트(ERD → Knowledge Graph → GraphRAG)의 현재 구현 상태를 분석하고,
프로덕션 수준(Docker 배포 + CI 통과 + rate limit 적용)으로 끌어올리기 위한
코드 레벨 수정 계획을 작성하세요.

## 채점 기준 (Rubric)

각 축 0-100점. 아래 기준에 따라 채점하세요:

| 등급 | 점수 | 의미 | 개선 항목 특성 |
|------|------|------|---------------|
| S | 90-100 | 프로덕션 배포 가능, 모범 사례 수준 | COULD만 존재 |

| C | 60-69 | 기능은 있으나 MUST 결함 존재 | MUST 2개 이상 |
| D | 50-59 | 기본 골격만 존재 | 주요 기능 미구현/불안정 |
| F | 0-49 | 미구현이거나 근본적 재설계 필요 | — |

감점은 반드시 코드 파일명:라인 또는 구체적 현상을 인용하여 근거를 대세요.

## 분석 축 (7축)

아래 7개 축에 대해 각각 **현재 수준 (0-100점)**, **확신도 (HIGH/MED/LOW)**, **근거**, **개선 항목**을 분석하세요.

### 축 1: 아키텍처 건전성
- 모듈 간 결합도 / 의존성 방향
- 관심사 분리 (라우터 ↔ 서비스 ↔ DB)
- 확장 패턴 일관성 (새 도메인 추가 시 변경점)
- 설정 관리 (환경변수, 시크릿)

### 축 2: 데이터 파이프라인 완성도
- DDL → 온톨로지 → KG 빌드 → 쿼리 전체 흐름의 안정성
- CSV / YAML 매핑 / 문서 수집 파이프라인 간의 일관성
- 에러 복구 / 재시도 전략
- 데이터 일관성 보장 (트랜잭션, 멱등성)

### 축 3: 쿼리 품질
- Mode A/B/C 각각의 정확도와 커버리지
- 라우팅 정확성 (규칙 vs LLM 폴백)
- 응답 지연 시간 (캐시 전략, 불필요한 LLM 호출)
- 다국어 / 다도메인 확장성

### 축 4: 테스트 & 품질 보증
- 테스트 커버리지 (유닛 / 통합 / E2E)
- 부정 테스트 / 엣지 케이스 비율
- 외부 의존 모킹 전략 (Neo4j, PG, LLM 없이 테스트 가능 여부)
- 회귀 테스트 자동화

### 축 5: 프로덕션 준비도
- 로깅 / 모니터링 / 알림
- 보안 (인증, 인가, 인젝션 방지, CORS)
- 성능 (동시 접속, 메모리, 대용량 DDL/문서)
- 배포 자동화 (Docker, CI/CD, 환경 분리)
- 데이터 백업 / 복구

### 축 6: 사용자 경험 (UX)
- 프론트엔드 완성도 (6개 페이지의 기능 / 디자인 일관성)
- 에러 메시지 / 로딩 상태 / 빈 상태 처리
- 워크플로우 자연스러움 (업로드 → 리뷰 → 빌드 → Q&A)
- 접근성 / 반응형

### 축 7: 확장성 & 유지보수
- 새 도메인 추가 시 코드 변경량
- API 버전 관리 전략
- 코드 문서화 (docstring, 타입 힌트)
- 기술 부채 (deprecated 코드, TODO, 하드코딩)

## 분석 규칙

1. **코드를 직접 읽고 판단하세요** — 파일 목록만 보고 추측하지 마세요
   - 각 축의 분석에서 최소 3개의 실제 파일경로:라인번호를 인용하세요
   - "~인 것으로 보인다" 같은 추측 표현 대신 "파일X:Y에서 확인됨" 형태로 작성하세요
   - 코드를 읽을 수 없는 환경이면 해당 축을 "검증 불가"로 표시하세요
2. **구체적으로 쓰세요** — "개선 필요" 대신 "graph_schema.py:175 _enrich_with_owl()에서 PG 커넥션을 매번 새로 열고 있어 커넥션 풀 활용 필요"
3. **기존 패턴을 존중하세요** — 프로젝트가 이미 사용하는 패턴(non-fatal, graceful degradation, 2-stage fallback)을 변경하지 마세요
4. **실현 가능성 우선** — 1인 풀스택 개발 환경을 고려하여 현실적인 범위만 제안하세요
5. **정량적 근거** — 가능한 한 LOC, 테스트 수, 엔드포인트 수 등 숫자로 뒷받침하세요
6. **v5.0+OWL 통합 고려** — 문서/NER/KG링킹/OWL이 서로 잘 맞물리는지 특히 분석하세요
7. **측정 불가 항목 처리** — 특정 축의 하위 항목이 해당 프로젝트에 적용되지 않거나 측정할 데이터가 없을 경우:
   - 해당 항목을 "N/A"로 표시하고 나머지 항목으로만 점수 산출
   - N/A 항목이 축의 50% 이상이면 해당 축을 "평가 불가"로 표시하고 종합 점수에서 제외
