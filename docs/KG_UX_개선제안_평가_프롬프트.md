# GraphRAG UX 개선 제안 — 심층 평가 및 대안 도출 프롬프트

> 이 프롬프트를 Claude, GPT 등 LLM에 그대로 복사하여 사용하세요.

---

## 프롬프트 시작

```
당신은 KG(Knowledge Graph) 시각화 전문가이자 B2B SaaS UX 아키텍트다.
아래에 (1) 프로젝트의 현재 구현 상태, (2) 기존에 도출된 3가지 UX 개선 제안이 있다.

당신의 임무는:
- 각 제안이 실제 구현되었을 때의 결과를 냉정하게 시뮬레이션하고
- 간과된 리스크, 과대평가된 기대효과, 숨겨진 기술적 함정을 짚어내고
- 기존 제안을 개선하거나, 완전히 새로운 더 나은 대안을 제시하는 것이다.

**중요: 기존 제안에 우호적일 필요가 없다. 아첨하지 말고 정직하게 평가하라.**

# 1. 프로젝트 현재 상태 (실제 코드 기반)

## 1-1. 아키텍처
- Frontend: React 19 + TypeScript + Vite + Tailwind CSS + shadcn/ui
- Graph: Cytoscape.js (react-cytoscapejs 래퍼), Canvas 렌더링
- Backend: FastAPI (Python) + Pydantic
- DB: Neo4j (KG), PostgreSQL (메타데이터), Redis (캐시)
- LLM: Claude (Anthropic API)
- 배포: Docker Compose (Neo4j, PG, Redis, Backend, Frontend/Nginx)

## 1-2. 현재 페이지 구조 (실제 코드 LOC 포함)

### ExplorePage.tsx (186줄) — 그래프 탐색
- Cytoscape 캔버스 + 3단 레이아웃 (좌: 필터사이드바, 중앙: 그래프, 우: 디테일패널)
- 500 노드 하드코딩 리밋으로 초기 로드
- 노드 클릭 → 속성 + 연결 노드 표시 (우측 패널)
- expandNode(): 1-hop 이웃 lazy loading
- 레이아웃: COSE(force-directed) 전환만 지원
- 검색: 클라이언트 사이드 substring 매칭 (display_name, id, label)
- 줌: 1.3x 배율 버튼 + fit-to-view

### QueryPage.tsx (291줄) — Q&A 채팅
- 모드 A(Template) / 모드 B(Local Search) 전환
- 5개 한국어 데모 질문 하드코딩
- 응답 표시: 모드 배지, 템플릿 ID, Cypher 쿼리, 실행 통계, evidence path, 캐시 여부
- **단일 턴 질의** (멀티턴 컨텍스트 없음)
- **텍스트 전용 결과** (그래프 시각화 연동 없음)

### use-graph-data.ts (228줄) — 데이터 훅
- fetchGraph(limit), fetchGraphStats(), fetchNeighbors(nodeId, depth)
- 노드/엣지 타입별 필터링 (Set<string> 기반)
- 검색 하이라이트: matched_nodeIds 세트 → 노란 보더
- 확장 시 중복 노드 자동 제거
- **캐싱 없음** (확장한 이웃 데이터 재요청 필요)

### graph-styles.ts (99줄) — 스타일
- 12종 엔티티별 고정 색상 (Customer=인디고, Product=에메랄드...)
- 선택 상태: 파란 보더 + 오버레이
- 엣지: 1.5px 회색, 삼각 화살표, 관계명 라벨
- **애니메이션/트랜지션 스타일 없음**
- **다크모드 없음**
- activeLabels 파라미터가 실제로는 미사용

### executor.py (147줄) — 쿼리 실행 (백엔드)
- **데모 전용**: 3개 하드코딩 질문만 처리
- 템플릿 기반 Cypher 채우기 (LLM 라우터 미구현)
- evidence path 포맷팅 (질문별 하드코딩)
- **임의 사용자 질의 미지원**
- **결과에 node ID 배열 미포함** (텍스트 답변만 반환)

### graph.py (246줄) — Graph API (백엔드)
- GET /graph/stats: 노드/엣지 타입별 카운트
- GET /graph/full?limit=500: 전체 그래프 (최대 5000)
- GET /graph/neighbors/{node_id}?depth=1: 이웃 탐색 (최대 3-hop)
- 복합 노드 ID: `{Label}_{id}` 형식
- **속성 기반 필터링 미지원**
- **랭킹/정렬 미지원**

## 1-3. 현재 시스템의 핵심 제약 (구현 사실)
1. QueryPage와 ExplorePage는 **완전히 독립된 페이지** (상태 공유 없음)
2. 쿼리 executor는 **3개 데모 질문 전용** (범용 자연어 → Cypher 파이프라인 미완성)
3. 그래프 스타일에 **애니메이션/glow 효과 없음** (정적 스타일만)
4. 백엔드 응답에 **관련 node ID가 포함되지 않음** (텍스트만)
5. 프론트엔드에 **WebSocket 미사용** (폴링 기반 비동기 작업만)
6. 멀티턴 대화 컨텍스트 **미지원**

# 2. 기존 3가지 UX 개선 제안

## 제안 1: Semantic Zoom Dashboard ("줌-어스" 패턴)
- **컨셉**: 줌 레벨에 따라 추상화 수준 자동 전환 (타입 개요 → 클러스터 → 개별 엔티티)
- **예상 구현**: Cytoscape compound nodes + 커스텀 줌 핸들러 + 백엔드 집계 API
- **난이도**: Medium
- **와우 팩터**: 수만 노드를 인지 부하 없이 탐색

## 제안 2: Conversational Graph Navigator ("대화형 그래프")
- **컨셉**: 좌측 그래프 + 우측 채팅 통합, 질문하면 그래프가 하이라이트
- **예상 구현**: QueryPage + ExplorePage 병합, LLM 응답에서 node ID 추출, 하이라이트 연동
- **난이도**: Medium-High
- **와우 팩터**: "질문하면 그래프가 살아난다" (Live Query Highlighting)

## 제안 3: Narrative Explorer ("스토리텔링 탐색")
- **컨셉**: KG에서 패턴/이상치 자동 감지 → 스토리 카드 → 단계별 가이드 탐색
- **예상 구현**: LLM이 Neo4j 통계 분석 → 스토리 JSON 생성 → StoryPlayer UI
- **난이도**: High
- **와우 팩터**: 비기술 사용자도 즉시 인사이트 획득

# 3. 평가 요청

## A. 각 제안별 "구현 후 현실" 시뮬레이션

각 제안에 대해, **실제로 구현을 완료하고 사용자 앞에 배포했다고 가정**했을 때:

### A-1. 기대 vs 현실 Gap 분석
각 제안마다 아래 테이블을 작성하라:

| 차원 | 제안서의 기대 | 구현 후 현실 예측 | Gap 원인 |
|------|-------------|-----------------|----------|
| 첫 사용자 반응 | | | |
| 반복 사용률 | | | |
| 인사이트 도출 빈도 | | | |
| 기술 부채 증가량 | | | |
| 유지보수 비용 | | | |

### A-2. 숨겨진 기술적 함정 (Hidden Pitfalls)
각 제안에서 **제안서가 언급하지 않았지만 구현 시 반드시 부딪힐 문제** 3가지 이상:
- 성능 병목 (어디서?)
- 에지 케이스 (어떤 데이터 패턴에서?)
- UX 단절점 (사용자 흐름이 끊기는 지점)

### A-3. 과대평가된 요소 vs 과소평가된 요소
각 제안에서:
- **과대평가**: 제안서가 장점으로 강조했지만 실제로는 효과가 미미할 요소
- **과소평가**: 제안서가 간과했지만 실제로는 핵심적인 난관

### A-4. 비용 대비 효과 (ROI) 순위
3가지 제안을 아래 기준으로 순위를 매기고 근거를 제시하라:
- **구현 비용**: 개발 시간 + 기술 부채 + 학습 비용
- **사용자 가치**: 실제 인사이트 도출 + 탐색 효율 + 비기술 사용자 접근성
- **데모 임팩트**: 멘토/투자자/고객에게 보여줄 때의 "와우" 효과
- **확장 가능성**: 이 기능 위에 다른 기능을 쌓을 수 있는 정도

## B. 기존 제안의 개선안

각 제안의 **핵심 컨셉은 유지하되**, 위에서 발견한 문제를 해결하는 개선 버전을 제시하라.
특히:
1. **현재 코드 상태에서 실현 가능한 범위**로 스코프를 조정 (executor.py가 데모 전용인 현실 반영)
2. **MVP 범위**를 명확히 정의 (최소한 무엇만 구현하면 핵심 가치가 전달되는가?)
3. **점진적 확장 경로**를 제시 (MVP → v1 → v2 단계별로 어떤 기능을 추가하는가?)

각 개선안에 대해:
```
### [제안명] 개선안
- **MVP 범위** (1주 이내 구현 가능): ...
- **MVP에서 제거한 것과 그 이유**: ...
- **MVP만으로도 전달되는 핵심 가치**: ...
- **v1 확장** (2-3주): ...
- **v2 확장** (1개월+): ...
- **개선된 와이어프레임** (ASCII):
```

## C. 제4의 대안 제안

기존 3가지 제안이 모두 **공유하는 맹점이나 전제**가 있다면 지적하고,
그 맹점을 해결하는 **완전히 다른 접근**을 1~2가지 제안하라.

고려할 관점:
- 기존 3가지는 모두 "시각화를 더 잘하자"에 초점 → 시각화 외의 가치 전달 방법은?
- 기존 3가지는 모두 "탐색 경험 개선"에 초점 → 탐색 이전/이후의 사용자 여정은?
- 기존 3가지는 모두 "실시간 인터랙티브"에 초점 → 비동기/배치/리포트 형태는?
- 기존 3가지는 모두 "단일 사용자"에 초점 → 협업/공유/주석 시나리오는?

각 대안에 대해:
```
### [대안명]
- **핵심 통찰** (기존 제안이 놓친 것): ...
- **컨셉**: ...
- **ASCII 와이어프레임**:
- **핵심 인터랙션 3가지**: ...
- **기존 3가지 대비 차별점**: ...
- **현재 코드베이스에서의 구현 난이도**: ...
- **MVP 범위**: ...
```

## D. 최종 추천: 실행 로드맵

위의 모든 분석을 종합하여, **현재 이 프로젝트가 다음 2개월간 실행해야 할 최적의 경로**를 제시하라.

형식:
```

| 평가 차원 | 현재 | Phase 2 후 | 변화 |
|-----------|------|-----------|------|
| ... | ... | ... | ... |

### Phase 3 (5-8주): [이름]

로드맵 작성 시 반드시 고려할 것:
1. executor.py가 데모 전용인 현실 → 범용 쿼리 파이프라인 구축이 선행되어야 하는가?
2. 프론트엔드 개선 vs 백엔드 파이프라인 완성 중 어디에 먼저 투자해야 하는가?

## 이 프롬프트의 설계 원리

| 설계 요소 | 이유 |
|-----------|------|
| **실제 코드 LOC + 제약사항 명시** | "executor.py가 3개 데모만 처리"라는 현실을 모르면 "LLM이 Cypher 생성" 같은 비현실적 제안이 나옴 |
| **"아첨하지 말라" 명시** | LLM은 기본적으로 제안을 긍정하려는 편향이 있음. 명시적으로 비판적 관점을 요구 |
| **기대 vs 현실 테이블** | 제안서의 장밋빛 전망과 구현 후 현실의 차이를 구조적으로 드러내기 위함 |
| **숨겨진 함정 3가지 이상** | 제안서가 언급 안 한 문제를 강제로 찾게 하여 blind spot 해소 |
| **MVP 범위 강제** | "전체 구현"이 아닌 "최소한 뭘 해야 가치 전달?"에 집중시킴 |
| **제4의 대안 + 공유 맹점** | 3가지 모두가 같은 프레임에 갇혀 있을 가능성을 열어둠 |
| **Phase별 로드맵** | 추상적 제안이 아닌 실행 가능한 주 단위 계획으로 마무리 |
| **백엔드 vs 프론트엔드 우선순위 질문** | UX 개선이 화려하지만 백엔드 파이프라인이 데모 수준인 현실을 직시하게 함 |
