# GraphRAG 프로젝트 구조 분석

> **작성일**: 2026-03-06
> **버전**: v1.0.0
> **코드베이스**: 백엔드 6,461줄 + 테스트 2,802줄 + 프론트엔드 5,606줄 = **14,869줄**

---

## 1. 프로젝트 개요

**한 줄 요약**: SQL DDL 파일을 업로드하면 Knowledge Graph를 자동 구축하고, 3가지 모드(A/B/W)로 질의응답하는 B2B SaaS 데모 제품.

```
DDL 파일 → ERD 파싱 → 온톨로지 생성 → KG 빌드(Neo4j) → Q&A (A/B/W 모드)
```

### 기술 스택

| 레이어 | 기술 |
|--------|------|
| Frontend | React 18 + TypeScript + Vite + Tailwind CSS + shadcn/ui |
| Backend | Python 3.12 + FastAPI + Pydantic v2 |
| Graph DB | Neo4j 5 Community (Docker / AuraDB) |
| Meta DB | PostgreSQL 16 (버전 관리, 빌드 이력) |
| LLM | OpenAI GPT-4o (주력) + Anthropic Claude (fallback) |
| Infra | Docker Compose (4 서비스) + GCP 배포 가능 |

## 3. 백엔드 아키텍처

### 3.1 데이터 플로우 (메인 파이프라인)

```
[1] DDL Upload            [2] Ontology Generate      [3] KG Build
━━━━━━━━━━━━━━━━━━━━━━━   ━━━━━━━━━━━━━━━━━━━━━━━    ━━━━━━━━━━━━━━━━━━━
.sql 파일                  ERDSchema                   OntologySpec + Data
    │                          │                            │
    ▼                          ▼                            ▼
ddl_parser/parser.py       ontology/fk_rule_engine.py  data_generator/generator.py
(sqlglot + regex)          (Stage 1: FK 규칙)          (시드 or CSV 데이터)
    │                          │                            │
    ▼                          ▼ (optional)                 ▼
ERDSchema                  ontology/llm_enricher.py    kg_builder/loader.py
(tables + FKs)             (Stage 2: Claude LLM)       (Neo4j MERGE)
                               │                            │
                               ▼                            ▼
                           OntologySpec                 Neo4j Graph
                           (nodes + rels)               (9 nodes, 12 rels)
```

```
[4] Query (A/B/W)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

A 모드 (템플릿):
  question → demo_cache (hit?) → router_rules → router_llm (fallback)
           → template_registry → slot_filler → Neo4j → 한국어 답변

B 모드 (로컬서치):
  question → extract_entity → build_subgraph_cypher → Neo4j
           → serialize_subgraph → GPT-4o → 자연어 답변

W 모드 (위즈덤):
  question → wisdom_router (intent 분류)
           → wisdom_collector (다중 Cypher 수집)
           → wisdom_prompts (직렬화)
           → GPT-4o (DIKW 분석)
           → 구조화된 4계층 응답
```

```
DDLParseError        → 422  DDL 파싱 실패
FileTooLargeError    → 413  파일 크기 초과
OntologyGenerationError → 422  온톨로지 생성 실패
LLMEnrichmentError   → 502  LLM API 실패 (non-fatal)
VersionNotFoundError → 404  버전 미존재
VersionConflictError → 409  승인된 버전 수정 시도
BuildJobNotFoundError → 404  빌드 잡 미존재
QueryRoutingError    → 422  쿼리 라우팅 실패
CypherExecutionError → 502  Neo4j 실행 실패
CSVValidationError   → 422  CSV 검증 실패
LocalSearchError     → 502  B/W 모드 LLM 실패
```

## 9. 설계 원칙 정리

| 원칙 | 적용 |
|------|------|
| **LLM은 보조, 규칙이 기본** | FK 규칙 엔진 → LLM은 보강/폴백만. API 키 없이도 핵심 동작 |
| **PG/LLM 장애는 non-fatal** | PG 연결 실패 → 로그 경고 후 인메모리 동작. LLM 실패 → 규칙 결과만 반환 |
| **워터폴 패턴** | 하드코딩 → 캐시 → 규칙 → LLM → 폴백 (loader.py name, query router) |
| **데모 우선** | demo_cache(A), 시드 데이터, Auto Demo — 외부 의존성 없이 시연 가능 |
| **하위 호환** | 새 필드는 default 값 설정, 기존 API 시그니처 유지 |
| **모듈별 독립성** | ddl_parser, ontology, query, wisdom — 각 모듈이 독립적으로 테스트 가능 |
