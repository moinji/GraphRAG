## 2. 강점 분석 (유지해야 할 것)

### 강점 1: 역할 지정이 태스크에 정확히 부합
> "당신은 B2B SaaS 아키텍트이자 시니어 풀스택 엔지니어입니다."

B2B SaaS + 풀스택이라는 두 축이 이 프로젝트의 실제 맥락(FastAPI + React + Neo4j)과 정확히 일치. "아키텍트"가 분석 태스크, "엔지니어"가 실행 계획 태스크를 각각 커버.

### 강점 2: 정량적 컨텍스트 제공이 우수
> "Backend 소스: ~14,700 LOC, 96개 파일 / 테스트: 40개 파일, ~593개 테스트"

LOC, 파일 수, 엔드포인트 수, 테스트 수 등 grounding 가능한 수치가 풍부하게 포함. LLM이 "소규모 사이드 프로젝트" vs "프로덕션 규모" 판단에 즉시 활용 가능.

### 강점 3: 출력 형식이 5-Part로 체계적
> "Part 1: 레이더 차트 → Part 2: 축별 상세 → Part 3: 우선순위 매트릭스 → Part 4: 실행 계획 → Part 5: 리스크"

분석 → 종합 → 실행으로 이어지는 논리적 흐름. 각 Part에 테이블 템플릿을 제공하여 형식 편차를 줄임. 특히 Part 3의 2×2 매트릭스와 Part 4의 Stage별 "수정 파일 / 테스트 수 / DoD"는 실행 가능성이 높은 형식.

### 강점 4: 분석 규칙 6조가 실용적
> "코드를 직접 읽고 판단하세요", "기존 패턴을 존중하세요", "1인 풀스택 개발 환경을 고려"

특히 규칙 3("기존 패턴을 존중하세요")과 규칙 4("1인 풀스택")는 LLM이 빠지기 쉬운 과도한 리팩터링 제안을 억제. 규칙 2의 `graph_schema.py:175` 같은 구체적 예시도 LLM에게 기대 수준을 명확히 전달.

### 강점 5: 구현 완료 기능 테이블
22개 기능 × 버전 × 상태의 테이블이 프로젝트 성숙도를 한눈에 보여줌. v0.1부터 v6.0까지의 진화 과정이 LLM에게 "이 프로젝트가 어디까지 왔는가"를 정확히 전달.

## 3. 치명적 결함 (즉시 수정 필요)

### 결함 1: 점수 기준이 정의되지 않음 — 재현 불가

**문제 현상**: 동일 프롬프트를 10번 실행하면 같은 축의 점수가 ±15점 이상 흔들릴 가능성이 높음. "아키텍처 건전성"에 65점을 줄지 80점을 줄지는 순전히 LLM의 기분(temperature)에 의존.

**원인 진단**:
> "각각 **현재 수준 (0-100점)**, **근거**, **개선 항목**을 분석하세요."

0-100점 스케일에 대한 채점 기준(rubric)이 전혀 없음. "등급: S(90+), A(80-89)..."는 점수→등급 변환 규칙일 뿐 "80점이 되려면 뭘 충족해야 하는가"를 정의하지 않음.

**수정안**:

Before:
```
아래 7개 축에 대해 각각 **현재 수준 (0-100점)**, **근거**, **개선 항목**을 분석하세요.
```

After:
```
아래 7개 축에 대해 각각 **현재 수준 (0-100점)**, **근거**, **개선 항목**을 분석하세요.

채점 기준 (rubric):
- 90-100 (S): 프로덕션 배포 가능, 모범 사례 수준. 개선 항목이 "nice to have"만 존재.
- 80-89 (A): 핵심 기능 안정, 구조적 결함 없음. 1-2개 SHOULD 수준 개선 존재.
- 70-79 (B): 작동하지만 구조적 약점 1-2개. 배포 전 반드시 보완 필요.
- 60-69 (C): 기능은 있으나 MUST 수준 결함 존재. 데모는 가능하나 프로덕션 불가.
- 50-59 (D): 기본 골격만 존재. 주요 기능 미구현 또는 불안정.
- 0-49 (F): 미구현이거나 근본적 재설계 필요.

감점은 반드시 코드 파일명:라인 또는 구체적 현상을 인용하여 근거를 대세요.
```

### 결함 2: Grounding 지시 부재 — 환각 위험

**문제 현상**: LLM이 실제 코드를 읽지 않고도 "아키텍처가 좋다", "테스트 커버리지가 높다"와 같은 일반론을 생성할 수 있음. 특히 컨텍스트에 이미 "96개 파일, 593개 테스트"가 주어져 있어 LLM이 코드를 읽은 것처럼 이 숫자를 반복하는 것만으로 그럴듯한 분석을 만들 수 있음.

**원인 진단**:
> "코드를 직접 읽고 판단하세요 — 파일 목록만 보고 추측하지 마세요"

이 지시는 좋은 의도이나 **검증 메커니즘이 없음**. LLM이 "코드를 읽었다"고 주장하면서 실제로는 컨텍스트 수치만 재활용하는 것을 막을 방법이 없음.

**수정안**:

Before:
```
1. **코드를 직접 읽고 판단하세요** — 파일 목록만 보고 추측하지 마세요
```

After:
```
1. **코드를 직접 읽고 판단하세요** — 파일 목록만 보고 추측하지 마세요
   - 각 축의 분석에서 최소 3개의 실제 파일 경로:라인번호를 인용하세요
   - "~인 것으로 보인다", "~일 가능성이 있다" 같은 추측 표현 대신
     "파일X:Y에서 확인됨" 형태로 작성하세요
   - 코드를 읽을 수 없는 환경이면 해당 축을 "검증 불가"로 표시하세요
```

### 결함 3: 엣지 케이스 미정의 — 측정 불가 상황에서 LLM이 임의 판단

**문제 현상**: 프론트엔드 테스트가 0개인 상황에서 "테스트 & 품질 보증" 축의 프론트엔드 부분을 어떻게 채점해야 하는지 지침이 없음. LLM마다 "FE 테스트 0개이므로 축 전체 30점"과 "BE 테스트가 충분하므로 70점" 사이에서 갈림.

**원인 진단**:
> 7개 축의 체크리스트에 "해당 없음" 또는 "측정 불가" 시 어떻게 처리할지 언급 없음

**수정안**:

Before: (분석 규칙 6조 다음에 추가)

After:
```
7. **측정 불가 항목 처리** — 특정 축의 하위 항목이 해당 프로젝트에 적용되지 않거나
   측정할 데이터가 없을 경우:
   - 해당 항목을 "N/A"로 표시하고 나머지 항목으로만 점수 산출
   - N/A 항목이 축의 50% 이상이면 해당 축을 "평가 불가"로 표시하고 종합 점수에서 제외
```

## 분석 축 (7축)

아래 7개 축에 대해 각각 **현재 수준 (0-100점)**, **확신도**, **근거**, **개선 항목**을 분석하세요. ◀ NEW: 확신도 추가

### 축 1: 아키텍처 건전성
- 모듈 간 결합도 / 의존성 방향
- 관심사 분리 (라우터 ↔ 서비스 ↔ DB)
- 확장 패턴 일관성 (새 도메인 추가 시 변경점)
- 설정 관리 (환경변수, 시크릿)

### 축 2: 데이터 파이프라인 완성도
- DDL → 온톨로지 → KG 빌드 → 쿼리 전체 흐름의 안정성
- CSV / YAML 매핑 / 문서 수집 파이프라인 간의 일관성
- 에러 복구 / 재시도 전략
- 데이터 일관성 보장 (트랜잭션, 멱등성)

### 축 3: 쿼리 품질
- Mode A/B/C 각각의 정확도와 커버리지
- 라우팅 정확성 (규칙 vs LLM 폴백)
- 응답 지연 시간 (캐시 전략, 불필요한 LLM 호출)
- 다국어 / 다도메인 확장성

### 축 4: 테스트 & 품질 보증
- 테스트 커버리지 (유닛 / 통합 / E2E)
- 부정 테스트 / 엣지 케이스 비율
- 외부 의존 모킹 전략 (Neo4j, PG, LLM 없이 테스트 가능 여부)
- 회귀 테스트 자동화

### 축 5: 프로덕션 준비도
- 로깅 / 모니터링 / 알림
- 보안 (인증, 인가, 인젝션 방지, CORS)
- 성능 (동시 접속, 메모리, 대용량 DDL/문서)
- 배포 자동화 (Docker, CI/CD, 환경 분리)
- 데이터 백업 / 복구

### 축 6: 사용자 경험 (UX)
- 프론트엔드 완성도 (6개 페이지의 기능 / 디자인 일관성)
- 에러 메시지 / 로딩 상태 / 빈 상태 처리
- 워크플로우 자연스러움 (업로드 → 리뷰 → 빌드 → Q&A)
- 접근성 / 반응형

### 축 7: 확장성 & 유지보수
- 새 도메인 추가 시 코드 변경량
- API 버전 관리 전략
- 코드 문서화 (docstring, 타입 힌트)
- 기술 부채 (deprecated 코드, TODO, 하드코딩)

### Part 2: 축별 상세 분석

각 축에 대해:
1. **현재 수준** (3문장 이내) ◀ NEW: "2-3줄" → "3문장 이내"

### Part 3: 우선순위 로드맵

모든 개선 항목을 아래 매트릭스로 분류:

```
               높은 임팩트          낮은 임팩트
낮은 노력    | Quick Wins         | Nice to Have
높은 노력    | Strategic Bets     | Deprioritize
```

### Part 4: 단계별 실행 계획

개선 항목을 3단계로 묶어 실행 계획 작성:

**Stage 1 (즉시, 1-2일)**: Quick Wins — 바로 효과가 나는 것
**Stage 2 (단기, 3-5일)**: Strategic — 아키텍처/품질 레벨업
**Stage 3 (중기, 1-2주)**: 프로덕션 준비 + 확장

각 Stage에 대해:
- 수정/생성할 파일 목록
- 예상 테스트 수
- 완료 기준 (Definition of Done)

### Part 5: 리스크 & 트레이드오프

각 Stage의:
- 기술적 리스크
- 기존 코드에 대한 영향 (breaking change 여부)
- 건너뛰었을 때의 비용

5. **정량적 근거** — 가능한 한 LOC, 테스트 수, 엔드포인트 수 등 숫자로 뒷받침하세요
