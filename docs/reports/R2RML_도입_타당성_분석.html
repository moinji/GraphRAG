R2RML 도입 타당성 분석

W3C RDB-to-RDF 매핑 표준을 GraphRAG 파이프라인에 도입해야 하는가?
GraphRAG v0.9.0 | 2026-03-06 | 6축 분석 + 4개 대안 비교

EXECUTIVE SUMMARY

결론: 미도입 NOT RECOMMENDED
  1. 근본적 불일치 — R2RML은 RDF 트리플 변환 표준이지만, 본 프로젝트 타겟은 Neo4j Property Graph다. 6주를 투자해도 사용자에게 보이는 가치가 없다.
  2. 범용화 병목 미해결 — 핵심 과제는 "도메인별 매핑 자동 생성"인데, R2RML은 매핑의 저장 형식만 바꿀 뿐 자동 생성 문제를 해결하지 못한다.
  3. 권고 대안: LLM 매핑 자동생성 강화 (안 C) — 기존 Stage 2 LLM enrichment를 확장하면 1.5~2주 투자로 4배 높은 ROI를 달성한다.

Section 02

범용화 기여도

핵심 질문: "도메인별 하드코딩" 문제를 R2RML이 해결하는가?

🔍
R2RML은 매핑을 선언적으로 표현하는 형식이지, 매핑을 자동으로 생성하는 엔진이 아닙니다. 문제의 본질은 "누가 매핑 규칙을 만드는가"이며, R2RML은 규칙의 저장 형식만 바꿉니다.

동일한 매핑, 다른 문법

현재 (Python) # 1줄이면 끝 TABLE_TO_LABEL = { "customers": "Customer" }
R2RML (Turtle) # 5줄 + URI 설계 필요 <#CustomerMapping> rr:logicalTable [ rr:tableName "customers" ]; rr:subjectMap [ rr:template "http://ex.com/{id}"; rr:class ex:Customer ].

최종 권고

R2RML 미도입 — 안 C(LLM 매핑 자동생성 강화) 추진
  • R2RML은 RDF 생태계 표준이며, Property Graph 타겟인 이 프로젝트에는 근본적으로 맞지 않는다. 6주 투자 대비 사용자에게 보이는 가치가 없고, 범용화 병목(매핑 자동 생성)을 해결하지 못한다.
  • 기존 Stage 2 LLM enrichment를 확장하여 DDL만으로 OntologySpec을 자동 생성하는 경로를 구축한다. 1.5~2주 투자로 도메인 무관 범용화를 달성하며, "AI 자동 매핑"은 강력한 시장 메시지가 된다.
  • R2RML에서 빌려올 유일한 가치는 "선언적 매핑 파일" 개념이다. Python 딕셔너리 하드코딩 대신 JSON/YAML 매핑 설정으로 외부화하는 것은 안 A의 일부로 0.5주면 가능하며, R2RML 전체를 도입할 필요가 없다.
