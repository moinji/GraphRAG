## Part 2: 축별 상세 분석

### 축 1: 아키텍처 건전성 — 78점 (B)

**현재 수준**
14개 라우터 → 서비스 → DB 3계층이 일관되게 적용됨. 미들웨어(CORS, Tenant, Exception Handler)가 `main.py`에서 선언적으로 등록됨. 96개 파일 / ~14,960 LOC 규모에서 관심사 분리가 잘 유지되고 있음.

**강점**
- 라우터가 얇음 — 비즈니스 로직은 서비스 레이어(`pipeline.py`, `service.py`, `local_search.py`)에 집중
- 17개 커스텀 Exception + 핸들러로 일관된 에러 응답 (`exceptions.py`)
- `pydantic-settings` 기반 설정 — 환경변수 오버라이드, `.env` 파일 지원
- Non-fatal graceful degradation 패턴이 LLM/PG/Neo4j 전체에 일관 적용

**약점**
- DB 접근 추상화 부재: `pg_client.py`, `neo4j_client.py`가 싱글턴 모듈 함수 형태 → DI(Dependency Injection) 불가, 테스트 시 모킹 어려움
- 인메모리 상태: `_csv_sessions` (`csv_import/parser.py:16`), `_jobs` (`kg_builder/service.py:27`) — 프로세스 재시작 시 소실, 다중 워커 불가
- LLM 프로바이더 분기가 서비스 코드에 산재 — 추상 LLM 인터페이스 없음 (OpenAI vs Anthropic 조건문이 `llm_enricher.py`, `router_llm.py`, `local_search.py` 등에 반복)
- `config.py`에 모델명 하드코딩 (`claude-sonnet-4-20250514`, `gpt-4o`) — 버전 업 시 코드 수정 필요

**구체적 개선 항목**
| 구분 | 항목 | 위치 |
|------|------|------|
| MUST | 인메모리 `_jobs` → PG 또는 Redis 이관 (다중 워커 지원) | `kg_builder/service.py:27` |
| SHOULD | LLM 프로바이더 추상 레이어 (Strategy 패턴) | `llm_enricher.py`, `router_llm.py`, `local_search.py` |
| SHOULD | DB 클라이언트 DI 전환 (`Depends()`) | `pg_client.py`, `neo4j_client.py` |
| COULD | 인메모리 `_csv_sessions` TTL 기반 정리 강화 | `csv_import/parser.py:16` |

### 축 2: 데이터 파이프라인 완성도 — 82점 (A)

**현재 수준**
DDL → FK 규칙 → LLM 보강 → OWL 추론 → KG 빌드 → 물리화까지 전체 파이프라인이 동작. CSV 임포트, YAML 매핑, 문서 수집(PDF/DOCX/MD/HTML), NER, KG 링킹까지 다양한 입력 경로를 지원.

**강점**
- MERGE 기반 멱등 로드 (`kg_builder/loader.py`) — 중복 실행 안전
- FK 규칙 엔진이 결정론적 — LLM 없이도 기본 온톨로지 생성 가능
- 2-stage NER (KG 매칭 → LLM 폴백) — KG 우선 매칭으로 불필요한 LLM 호출 최소화
- OWL 추론이 기존 파이프라인에 비침투적 (`enable_owl: bool = False` 기본)
- 실패 시 자동 롤백: KG 빌드 실패 → `MATCH (n) DETACH DELETE n`

**약점**
- 다단계 로드(노드 → 관계 → 물리화)가 원자적이지 않음 — 중간 실패 시 부분 로드 상태 가능
- 문서 임베딩 생성이 동기 (`embedding_service.py`) — 대량 문서 시 블로킹
- `tenant_id=None`일 때 테넌트 격리 쿼리가 `WHERE n.tenant_id IS NULL`이 아닌 무시될 수 있음
- CSV 세션이 인메모리 — 대용량 파일 업로드 시 메모리 압박 가능

**구체적 개선 항목**
| 구분 | 항목 | 위치 |
|------|------|------|
| MUST | KG 로드 트랜잭션 래핑 (Neo4j explicit tx) | `kg_builder/loader.py` |
| SHOULD | 문서 임베딩 비동기 배치 처리 (BackgroundTask or 큐) | `documents/embedding_service.py` |
| SHOULD | tenant_id=None 방어 (빈 문자열 또는 기본값 할당) | `kg_builder/service.py` |
| COULD | CSV 세션 파일 기반 or Redis 이관 | `csv_import/parser.py` |

### 축 3: 쿼리 품질 — 75점 (B)

**현재 수준**
Mode A (Cypher 템플릿 13개), Mode B (서브그래프 로컬 검색), Mode C (벡터+KG 하이브리드) 3가지 모드 지원. 하이브리드 라우터(규칙 12패턴 → LLM 폴백). 52개 골든 QA 페어로 평가 체계 존재.

**강점**
- 3단계 라우팅: 규칙 → LLM → unsupported (비용 최적)
- 데모 캐시 (`demo_cache.py`) — 시드 데이터 기반 확정 답변으로 데모 안정성 보장
- Mode C가 벡터 유사도 + KG 구조 + 키워드 리랭킹 3중 신호 결합
- 슬롯 필러의 인젝션 방지 (`slot_filler.py`) — Cypher 인젝션 차단
- `_CLOSURE` 관계를 통한 OWL 추론 결과의 쿼리 시점 활용

**약점**
- 규칙 라우터 패턴이 한국어 + 영어 하드코딩 — 동의어/유사 표현 미지원 (예: "주문" vs "오더" vs "order")
- LLM 리랭커가 스텁 상태 (`reranker.py`) — keyword 점수만 실제 동작, LLM 리랭킹은 비용 이슈로 비활성
- Mode B가 엔터티 추출 실패 시 aggregate 폴백만 존재 — 중간 복잡도 질문에 약함
- Cypher 템플릿 13개로는 도메인 확장 시 커버리지 부족 — 동적 Cypher 생성 필요
- 응답 지연 측정은 있으나 캐시 전략이 데모 캐시에 한정 (일반 쿼리 캐시 없음)

**구체적 개선 항목**
| 구분 | 항목 | 위치 |
|------|------|------|
| MUST | 규칙 라우터 동의어 사전 추가 (Synonym map) | `query/router_rules.py` |
| SHOULD | 동적 Cypher 생성 (LLM → Cypher, schema-aware) | `query/` (신규) |
| SHOULD | 일반 쿼리 결과 캐시 (LRU or Redis, TTL 기반) | `query/pipeline.py` |
| COULD | LLM 리랭커 활성화 (비용 임계값 기반 선택) | `query/reranker.py` |

### 축 4: 테스트 & 품질 보증 — 80점 (A)

**현재 수준**
39개 테스트 파일, ~508개 테스트 함수. conftest.py에서 ecommerce_erd 등 공유 fixture 제공. 외부 의존(Neo4j, PG, LLM) 모킹으로 단독 테스트 가능. OWL 43개 테스트 포함.

**강점**
- 높은 커버리지: 파서(16) + 온톨로지(16) + 쿼리(19) + 로컬 검색(15) + QA 평가(10) + CSV(16) + 문서(74) + OWL(43) 등 영역별 분포 균형
- conftest.py fixture로 반복 가능한 테스트 환경
- 부정 테스트 존재: `test_unknown_axiom_type`, `test_property_chain_invalid_target`, empty/None 입력 등
- 골든 데이터 기반 회귀 테스트 (`test_regression.py`, 10 케이스)

**약점**
- E2E 통합 테스트가 `test_e2e_integration.py` 1개 파일에 집중 — API 레벨 E2E 부족
- Neo4j/PG 실제 연결 테스트가 skip 처리 — CI에서 docker-compose 기반 통합 테스트 미구축
- 프론트엔드 테스트 0개 — React 컴포넌트/API 호출 테스트 없음
- 성능/부하 테스트 없음 — 대용량 DDL, 동시 쿼리 시나리오 미검증
- 일부 테스트가 Education DDL 파일 의존 (`pytest.skip` 처리)

**구체적 개선 항목**
| 구분 | 항목 | 위치 |
|------|------|------|
| MUST | docker-compose 기반 통합 테스트 CI 워크플로우 | `.github/workflows/` (신규) |
| SHOULD | API 레벨 E2E 테스트 (TestClient, 전체 플로우) | `tests/test_api_e2e.py` (신규) |
| SHOULD | 프론트엔드 유닛 테스트 (Vitest + Testing Library) | `frontend/__tests__/` (신규) |
| COULD | 부하 테스트 (locust or k6, 기본 시나리오) | `tests/perf/` (신규) |

### 축 5: 프로덕션 준비도 — 52점 (D)

**현재 수준**
개발 환경에서 안정적으로 동작하나, 프로덕션 배포에 필요한 인프라가 대부분 미구축. docker-compose.yml(Neo4j)은 있으나 앱 컨테이너 미포함. CI/CD 파이프라인 없음. 모니터링/알림 없음.

**강점**
- 구조화된 로깅 (`logging` 모듈, 모든 서비스에 `logger = logging.getLogger(__name__)`)
- `.env` 기반 설정 분리 + `.gitignore`에 `.env` 등록
- CORS 미들웨어 설정 (`main.py`)
- 앱 API 키 인증 (`app_api_key`, `tenant_keys`) 기본 지원
- 비밀번호 기본값이 config에 있으나 `.env` 오버라이드 의도 (개발용)

**약점**
- Dockerfile 미존재 — 컨테이너 빌드 불가
- CI/CD 파이프라인 없음 (GitHub Actions 등)
- 헬스체크가 단순 (`/health` 200) — Neo4j/PG 연결 상태 미확인
- 인증이 단일 API 키 수준 — JWT/OAuth 미지원, RBAC 없음
- rate limiting 없음 — LLM 호출 비용 폭주 가능
- 프로덕션 로깅 포맷(JSON structured) 미설정
- 데이터 백업/복구 전략 없음
- HTTPS/TLS 설정 없음 (리버스 프록시 의존 가정)

**구체적 개선 항목**
| 구분 | 항목 | 위치 |
|------|------|------|
| MUST | Dockerfile (backend + frontend) + docker-compose 통합 | `Dockerfile`, `docker-compose.yml` |
| MUST | 헬스체크 강화 (Neo4j ping + PG ping + 디스크 여유) | `routers/health.py` |
| MUST | rate limiting (slowapi or 커스텀 미들웨어) | `main.py` |
| SHOULD | GitHub Actions CI (lint + test + build) | `.github/workflows/ci.yml` |
| SHOULD | JSON structured logging (production 모드) | `config.py`, `main.py` |
| COULD | JWT 인증 + RBAC | `middleware/auth.py` (신규) |
| COULD | Prometheus 메트릭 + Grafana 대시보드 | `routers/metrics.py` 확장 |

### 축 6: 사용자 경험 (UX) — 72점 (B)

**현재 수준**
6개 페이지 (Upload, Review, Query, Documents, Graph Visualization, Wisdom). React + TypeScript + Tailwind + shadcn/ui 기반. SSE 스트리밍, Force/Radial 그래프 시각화 지원.

**강점**
- 워크플로우가 자연스러움: Upload → Review → Build KG → Q&A 순차 진행
- shadcn/ui 기반 일관된 디자인 시스템
- SSE 스트리밍으로 실시간 진행 상태 표시
- 그래프 시각화 2가지 레이아웃 (Force, Radial)
- Mode A/B/C 토글, 데모 질문 버튼, 캐시 배지 등 UX 세부 장치

**약점**
- 에러 상태 처리가 `alert()` 또는 콘솔 로그에 의존하는 곳이 있음
- 로딩 스피너/스켈레톤이 일관되지 않음
- 빈 상태(empty state) 처리 미흡 — 데이터 없을 때 안내 부족
- 접근성(a11y): aria 속성 부족, 키보드 내비게이션 미검증
- 반응형 디자인이 데스크톱 우선 — 모바일에서 그래프 시각화 사용 어려움
- 프론트엔드 상태 관리가 prop drilling 위주 — 전역 상태 관리(Zustand 등) 없음

**구체적 개선 항목**
| 구분 | 항목 | 위치 |
|------|------|------|
| MUST | 에러 바운더리 + Toast 알림 통합 | `App.tsx`, 각 페이지 |
| SHOULD | 빈 상태/로딩 상태 컴포넌트 표준화 | `components/ui/` |
| SHOULD | 전역 상태 관리 도입 (Zustand or Jotai) | `frontend/src/store/` (신규) |
| COULD | 접근성 감사 + aria 속성 보완 | 전체 컴포넌트 |
| COULD | 반응형 그래프 시각화 (모바일 대응) | `GraphVisualization` 컴포넌트 |

### 축 7: 확장성 & 유지보수 — 76점 (B)

**현재 수준**
새 도메인(교육/보험) 추가가 비교적 쉬움 — `domain_hints.py`에 힌트, `axiom_registry.py`에 axiom 추가로 대응. 타입 힌트 전수 적용. TODO/FIXME 0개로 기술 부채 관리 양호.

**강점**
- 도메인 확장 패턴 표준화: `DomainHint` → `AxiomSpec` → 템플릿 → 골든 데이터
- TODO/FIXME/HACK 0개 — 기술 부채 인지 관리 우수
- Python 타입 힌트 전수 적용, Pydantic 모델로 스키마 검증
- 프론트엔드 TypeScript 타입이 백엔드 모델과 1:1 대응 (`types/ontology.ts`)
- 모듈별 독립 테스트 가능 (공유 fixture)

**약점**
- API 버전 관리 없음 — 현재 `/api/v1/` 경로이나 v2 전략 미정
- Docstring이 일부 핵심 함수에만 존재 — 공개 API 함수 중 30~40% 누락 추정
- `config.py` 모델명 하드코딩 — 모델 버전업 시 코드 수정 필요
- 새 도메인 추가 시 변경점이 4~5개 파일에 분산 (hints, axioms, golden, templates, rules)
- 프론트엔드 타입 수동 동기화 — 자동 생성 미적용 (openapi-typescript 등)

**구체적 개선 항목**
| 구분 | 항목 | 위치 |
|------|------|------|
| SHOULD | OpenAPI → TypeScript 자동 생성 (openapi-typescript) | `frontend/scripts/` (신규) |
| SHOULD | 도메인 추가 체크리스트/가이드 문서 | `docs/DOMAIN_GUIDE.md` (신규) |
| COULD | API v2 네임스페이스 전략 설계 | `main.py`, `routers/` |
| COULD | 핵심 서비스 함수 docstring 보강 | `pipeline.py`, `service.py` 등 |

## Part 5: 리스크 & 트레이드오프

### Stage 1 리스크

| 리스크 | 영향 | 대응 |
|--------|------|------|
| 동의어 사전 오탐 (과도한 매칭) | 잘못된 라우팅 | 동의어는 보수적으로 시작, 골든 데이터로 검증 |
| Toast 라이브러리 추가로 번들 크기 증가 | ~5KB (무시 가능) | sonner는 ~2KB gzip |
| **기존 코드 breaking change**: 없음 | | |
| **건너뛰었을 때 비용**: 데모 시 DB 장애가 감지 안 됨, 프론트 에러 무시됨 | | |

### Stage 2 리스크

| 리스크 | 영향 | 대응 |
|--------|------|------|
| _jobs PG 이관 시 기존 진행 중 빌드 소실 | 빌드 재시작 필요 | 배포 시 빌드 큐 비움 확인 |
| rate limiting이 정상 사용자 차단 | UX 저하 | 개발 환경에서는 비활성, 임계값 넉넉하게 (100 req/min) |
| Dockerfile 최적화 부족 시 이미지 2GB+ | 배포 느림 | multi-stage 빌드, .dockerignore |
| **기존 코드 breaking change**: `_jobs` API 호환 유지 필요 | | PG 이관 시 동일 인터페이스 유지 |
| **건너뛰었을 때 비용**: 컨테이너 배포 불가, 워커 스케일아웃 불가, LLM 비용 통제 불가 | | |

### Stage 3 리스크

| 리스크 | 영향 | 대응 |
|--------|------|------|
| 동적 Cypher 생성의 SQL 인젝션 유사 위험 | 보안 | 파라미터 바인딩 강제, 화이트리스트 검증 |

| **건너뛰었을 때 비용**: 새 LLM 모델 전환에 3-4개 파일 수동 수정, 도메인 확장 진입 장벽 높음 | | |
